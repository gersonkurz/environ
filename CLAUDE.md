# CLAUDE.md — Environ

## What this is
A Windows 11 environment variable editor. WinUI 3, C++/WinRT, C++20. No C#. No vcpkg. No exceptions.

## Build
```
cmake -B build -S . --preset windows
cmake --build build
```
Windows App SDK comes via NuGet. Everything else is a git submodule under `extern/`.

## Architecture — read this before touching anything
Two layers. Keep them separate.

**`src/core/`** — plain C++20. No WinRT headers. No Windows App SDK. This code must remain portable enough that a unit test runner can link it without a WinUI runtime present.

**`src/ui/`** — C++/WinRT + XAML. Calls into core. Does not contain business logic.

If you find yourself putting path validation in a `.xaml.cpp` file, you're doing it wrong.

## Code style
- C++20: use concepts, ranges, `std::format`, structured bindings where they make the code clearer — not to show off
- Uniform initializers everywhere. `int x{42}`, not `int x = 42`
- Use `gersonkurz/pnq` where it covers the need. Don't reimplement what's already there
- No raw owning pointers. `std::unique_ptr` or WinRT smart pointers depending on context
- Prefer `std::wstring` at the boundary between core and UI; don't let `winrt::hstring` leak into core
- spdlog for all logging. No `OutputDebugString` spam
- If something needs elevation, it goes through the established UAC boundary — do not silently fail or silently succeed

## What "done" means for a task
- Compiles without warnings (`/W4`)
- Follows the two-layer rule above
- Dark mode doesn't look broken
- Doesn't crash when run unelevated and the user touches a machine variable

## Do not
- Add dependencies without asking. The dependency list is deliberately minimal
- Introduce vcpkg. Not negotiable
- Write C++/WinRT component boilerplate by hand if codegen can do it — use the tooling
- Leave TODO comments without a GitHub issue reference
- Add "Generated by Claude" or any AI attribution to git commits. Commit messages are plain and descriptive — what changed and why, nothing else
